28 // CompositeInstructions are Instructions that contain further Instructions (which
29 // of course can be other CompositeInstructions). This forms the familiar tree, or
30 // composite pattern, where nodes are CompositeInstructions and leaves are concrete
31 // Instruction instances. CompositeInstructions are also Persistable, meaning a
32 // CompositeInstruction can be persisted (read) from an output (input) stream.
33 
34 // Adding Instructions:
35 // Implementations of CompositeInstruction are responsible for providing
36 // an appropriate container for Instructions. When adding parameterized
37 // Instructions (which may contain a string variable),
38 // implementations must verify that this CompositeInstruction exposes
39 // a variable string that corresponds to that same variable name.
40 // i.e. adding Ry(theta) to CompositeInstruction f, f must expose a variable
41 // with name theta.
42 
43 // CompositeInstructions can exist without children. These are considered
44 // terminating nodes in Instruction tree, and represent composite
45 // Instructions that can not be known at compile time, but are
46 // parameterized by a map of options. Clients can query if the
47 // CompositeInstruction hasChildren(), and if not, expand the CompositeInstruction to a
48 // list of its constituent children at runtime given a valid
49 // set of options. i.e., a CompositeInstruction that can generate a UCCSD circuit
50 // for gate model quantum computing needs to know the number of
51 // qubits and number of electrons, which may be unknown at CompositeInstruction
52 // build time. Later, clients can invoke f->expand({{"ne",2},{"nq",4}})
53 // to expand this CompositeInstruction and make it a composite node instead of a
54 // terminating node. In this way, developers can contribute dynamic
55 // runtime parameterized Instructions.
56 
57 // Note, expanding a CompositeInstruction node should expand all sub-nodes.
58 
59 // These runtime, expanded CompositeInstructions are intended to be provided
60 // through the xacc ServiceRegistry.
61 
62 // The operator()() evaluate method assumes that clients provide
63 // a vector of doubles, where the ith double corresponds to the
64 // ith variable.
65 
66 // Oftentimes, collections of instructions may have a coefficient
67 // associated with them, e.g. a term in a Hamiltonian may have a weight,
68 // but we map that term to a set of measurement instructions. Therefore,
69 // we also expose a coefficient complex value on the CompositeInstruction.
