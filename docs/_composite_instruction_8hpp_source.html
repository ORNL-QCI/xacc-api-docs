28 // CompositeInstructions are Instructions that contain further Instructions
29 // (which of course can be other CompositeInstructions). This forms the familiar
30 // tree, or composite pattern, where nodes are CompositeInstructions and leaves
31 // are concrete Instruction instances. CompositeInstructions are also
32 // Persistable, meaning a CompositeInstruction can be persisted (read) from an
33 // output (input) stream.
34 
35 // Adding Instructions:
36 // Implementations of CompositeInstruction are responsible for providing
37 // an appropriate container for Instructions. When adding parameterized
38 // Instructions (which may contain a string variable),
39 // implementations must verify that this CompositeInstruction exposes
40 // a variable string that corresponds to that same variable name.
41 // i.e. adding Ry(theta) to CompositeInstruction f, f must expose a variable
42 // with name theta.
43 
44 // CompositeInstructions can exist without children. These are considered
45 // terminating nodes in Instruction tree, and represent composite
46 // Instructions that can not be known at compile time, but are
47 // parameterized by a map of options. Clients can query if the
48 // CompositeInstruction hasChildren(), and if not, expand the
49 // CompositeInstruction to a list of its constituent children at runtime given a
50 // valid set of options. i.e., a CompositeInstruction that can generate a UCCSD
51 // circuit for gate model quantum computing needs to know the number of qubits
52 // and number of electrons, which may be unknown at CompositeInstruction build
53 // time. Later, clients can invoke f->expand({{"ne",2},{"nq",4}}) to expand this
54 // CompositeInstruction and make it a composite node instead of a terminating
55 // node. In this way, developers can contribute dynamic runtime parameterized
56 // Instructions.
57 
58 // Note, expanding a CompositeInstruction node should expand all sub-nodes.
59 
60 // These runtime, expanded CompositeInstructions are intended to be provided
61 // through the xacc ServiceRegistry.
62 
63 // The operator()() evaluate method assumes that clients provide
64 // a vector of doubles, where the ith double corresponds to the
65 // ith variable.
66 
67 // Oftentimes, collections of instructions may have a coefficient
68 // associated with them, e.g. a term in a Hamiltonian may have a weight,
69 // but we map that term to a set of measurement instructions. Therefore,
70 // we also expose a coefficient complex value on the CompositeInstruction.
